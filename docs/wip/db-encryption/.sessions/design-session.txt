  CAPY_DB_KEY environment variable is the single source of the passphrase. It's
  read once at DB open time ‚ÄĒ not stored in config files, not persisted
  anywhere.

  Startup flow: openDB() in store.go reads os.Getenv("CAPY_DB_KEY"). If empty,
  return an error ‚ÄĒ capy refuses to start. The error message is explicit:
  "CAPY_DB_KEY environment variable is required (see: capy encrypt --help)".

  Applying the key: Depends on the driver PoC result. Two paths:

  - PRAGMA path (SQLCipher, options 1/3): After sql.Open(), immediately execute
  PRAGMA key = ? with the passphrase as parameterized argument. This must be the
   first statement ‚ÄĒ before schema init, migrations, or any other pragma. The
  existing openDB() sequence becomes: open ‚Üí PRAGMA key ‚Üí PRAGMA mmap_size ‚Üí
  schema ‚Üí migrations ‚Üí prepared statements.
  - URI path (sqlite3mc, option¬†2): Append &cipher=chacha20&key=<passphrase> to
  the DSN string. No post-open PRAGMA needed. Cleaner, but requires URL-encoding
   the passphrase.

  Checkpoint connections also need the key. Both checkpoint() (internal) and
  Checkpoint() (public) open fresh sql.Open() connections. These must apply the
  same key ‚ÄĒ either via PRAGMA or URI. The passphrase is read from the env var
  again (not cached in the struct), so rotating the env var between runs works
  naturally.

  Validation on open: After applying the key, execute a canary query (SELECT 
  count(*) FROM sqlite_master) to detect wrong-key errors immediately rather
  than failing on the first real operation. SQLCipher/sqlite3mc return a generic
   error on wrong key ‚ÄĒ the wrapper translates this to "wrong passphrase or 
  corrupted database".

‚óŹ Section 3: capy encrypt Command

  New CLI command in cmd/capy/encrypt.go. Handles two scenarios: initial
  encryption (unencrypted ‚Üí encrypted) and key rotation (old key ‚Üí new key).
  Both use sqlcipher_export (or sqlite3mc equivalent) to create a new encrypted
  DB from the source.

  Prompt flow:

  1. Prompt for current DB passphrase (terminal stdin, no echo). Empty means the
   DB is currently unencrypted.
  2. Read new passphrase from CAPY_DB_KEY if set. If not set, prompt
  interactively (with confirmation prompt).
  3. If new passphrase is under 32 characters, print a warning: "Warning: 
  passphrase is under 32 characters. Short passphrases are easier to 
  brute-force." Proceed anyway.

  Encryption process:

  1. Resolve DB path from config (same logic as capy checkpoint ‚ÄĒ config.Load +
  ResolveDBPath).
  2. Verify DB file exists. Error if not.
  3. Open source DB with old key (empty key = open unencrypted). Run canary
  query to verify.
  4. Create temp target: <dbpath>.enc.tmp.
  5. Attach target with new key: ATTACH DATABASE '<target>' AS target KEY 
  '<new-key>'.
  6. Export: SELECT sqlcipher_export('target').
  7. Detach and close.
  8. Back up original: rename <dbpath> ‚Üí <dbpath>.bak.
  9. Rename temp to final: <dbpath>.enc.tmp ‚Üí <dbpath>.
  10. Remove WAL/SHM sidecars from original (they're incompatible with the new
  encrypted DB).
  11. Verify: reopen the new DB with the new key, run canary query.
  12. Print success: "Encrypted: <dbpath>. Backup at <dbpath>.bak".

  If any step fails after the backup rename, the .bak file preserves the
  original. The temp file is cleaned up on error.

  Key rotation is identical ‚ÄĒ step 3 opens with the old key instead of no key.
  The rest is the same.

  Does this look right?

  .capy/knowledge.db-wal
  .capy/knowledge.db-shm

  These two lines go after line 38. The sidecars are always transient ‚ÄĒ capy 
  checkpoint flushes them into the main file.

  Pre-commit hook update. The existing hook in internal/platform/ calls capy 
  checkpoint as a safety net. It gains one check: if the DB file is staged for
  commit, verify it's encrypted by checking the file header. An unencrypted
  SQLite DB starts with "SQLite format 3\000" (16 bytes). An encrypted DB has
  random bytes at the start. If the header matches the plaintext signature, the
  hook rejects the commit with: "Refusing to commit unencrypted knowledge.db. 
  Run 'capy encrypt' first.".

  This is the safety net against accidentally committing before encrypting ‚ÄĒ
  since the DB path is opt-in for in-repo storage, the check only fires when the
   file is actually staged, not on every commit.

‚óŹ Section 5: ADR-019 ‚ÄĒ Supersedes ADR-015

  ADR-015 decided "never track knowledge.db in git" due to WAL/SHM sidecar
  corruption on branch switches. That constraint is relaxed ‚ÄĒ not removed ‚ÄĒ by
  this feature.

  New decision: Committing knowledge.db is safe when two conditions are met:

  1. DB is encrypted. The pre-commit hook verifies this by checking the file
  header. Unencrypted DBs are still rejected.
  2. WAL is flushed. capy checkpoint must run before commit. The pre-commit hook
   already ensures this (existing behavior). After checkpoint, the WAL/SHM
  sidecars are empty or absent, so git only tracks the self-contained main file.

  The WAL corruption problem from ADR-015 hasn't gone away ‚ÄĒ it still applies if
   sidecars desync on branch switch. But capy checkpoint before commit
  eliminates sidecars, and capy recreates them on next startup. The risk surface
   is: if someone opens the DB outside capy while switching branches. The
  pre-commit hook is the guardrail.

  What ADR-019 does NOT change:

  - Default DB path stays XDG (~/.local/share/capy/<hash>/knowledge.db). Users
  opt into in-repo storage via store.path = ".capy/knowledge.db" in config.
  - The encryption requirement is unconditional ‚ÄĒ CAPY_DB_KEY is mandatory
  regardless of where the DB lives. Even XDG-path DBs are encrypted. The
  "committable" story is a consequence of encryption, not its primary purpose.
  - ADR-015's recommendation to not track the DB still applies to unencrypted
  DBs. ADR-019 narrows the prohibition: unencrypted = don't commit, encrypted +
  checkpointed = safe to commit.

  Status: Supersedes ADR-015 (ADR-015 becomes Superseded by ADR-019).
