‚Źļ Contradiction 12: Sandbox environment strategy ‚ÄĒ denylist vs allowlist

  - V1 design (section 13.1, lines 809-839) and implementation: describes a
  denylist approach ‚ÄĒ start with the full parent environment, strip ~50
  dangerous vars, apply sandbox overrides
  - V2 implementation (section 3.9, lines 730-771): describes an allowlist
  approach ‚ÄĒ start with a minimal set (PATH, HOME, USER), add specific
  passthrough vars (auth tokens, infra, network, config), add forced vars
  - context-mode source (executor.ts lines 319-449): uses the denylist approach
  ‚ÄĒ iterates process.env, skips entries in the DENIED set and BASH_FUNC_*
  prefixed vars, then applies overrides

  This is a meaningful security architecture decision, not just a formatting
  difference.

  Denylist (V1/context-mode): Start with everything, remove known-bad. Better
  compatibility ‚ÄĒ user's project-specific env vars (database URLs, custom API
  keys, service endpoints) work automatically. Risk is missing a new dangerous
  var, but context-mode's list is well-researched and CVE-backed.

  Allowlist (V2): Start with nothing, add known-good. Tighter by default, but
  breaks when users depend on env vars not in the curated list. V2's passthrough
