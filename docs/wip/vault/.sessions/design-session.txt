  Why not a separate project

  1. Parser duplication. The JSONL parsing in internal/session/ is 470 lines of
  battle-tested code handling progressive snapshots, subagent discovery, and
  edge cases. A separate project would either duplicate this or import capy as a
   library ‚ÄĒ both worse than just extending capy.
  2. Store reuse. Capy's encrypted SQLite + FTS5 stack is exactly what vault
  needs. claude-vault literally built the same thing from scratch in Rust.
  3. Hook system. Capy already installs hooks into Claude Code. Adding
  PreCompact and SessionEnd hooks for auto-archival is an incremental change,
  not a new system.
  4. Single DB key. Encryption is mandatory in capy. A separate project means a
  separate key or sharing CAPY_DB_KEY ‚ÄĒ awkward either way.
  5. Namespace fit. capy vault import, capy vault search etc. feel natural next
  to capy sweep, capy cleanup, capy doctor.

  Key design decision: separate DB vs. extending existing

  The one thing I'd push back on is whether vault should share the same SQLite
  DB as the FTS knowledge store. I'd argue separate DB file, same encryption 
  infrastructure:

  - The knowledge DB has retention/cleanup policies that would fight with
  "archive everything forever"
  - Verbatim JSONL storage bloats differently than FTS chunks
  - Different backup/export needs
  - But same CAPY_DB_KEY, same store package patterns, same config.ResolveDBPath
   convention

  This is analogous to how claude-vault puts its vault.db in
  ~/.local/share/claude-vault/ ‚ÄĒ separate from the working knowledge.

  What I'd borrow from each inspiration project

  - claude-vault: Hook-driven auto-archive (PreCompact is critical ‚ÄĒ captures
  data before /compact destroys it), FTS5 schema design
  - claude-history: TUI with fuzzy search, fork/resume from viewer, filtering
  out noise sessions
  - cc-sessions (your gist): Incremental index via flat metadata, resume by
  partial ID
  - pi-share-hf: Multi-stage redaction pipeline for sharing, TruffleHog-style
  secret scanning

  breakdown?

‚óŹ Section 4: FTS Scanner

  The scanner is a single-pass JSONL reader in internal/vault/scanner.go that
  extracts searchable text for the FTS5 index. It defines its own minimal JSON
  wire types (~20 lines) decoupled from internal/session/parse.go.

  What it extracts per turn:
  1. All "text" values from content blocks (user and assistant messages)
  2. Tool use names (e.g. "Bash", "Read", "Edit") ‚ÄĒ so you can search "which
  session used Bash heavily"
  3. Tool input summaries ‚ÄĒ file paths from Read/Edit, commands from Bash,
  queries from search tools
  4. Subagent descriptions and types

  What it skips:
  - Base64 and binary content (images, compressed state)
  - System-reminder tags and their contents
  - Progressive snapshot duplicates ‚ÄĒ tracked via a seen map keyed by message.id
   + block index. If a content block was already processed under the same
  message ID, skip it.
  - JSON structural noise (keys, braces, type markers)

  Output: For each turn, the scanner produces a ScanResult struct:

  type ScanResult struct {
      TurnIndex   int
      Role        string    // "user", "assistant", "system"
      SubagentID  string    // "" for main session
      ContentText string    // extracted searchable text
      Timestamp   time.Time
  }

  The import pipeline calls the scanner once per session file (and once per
  subagent file), then inserts one FTS row per ScanResult:

  INSERT INTO vault_fts (content_text, session_uuid, subagent_id, turn_index,
  role)
  VALUES (?, ?, ?, ?, ?)

  Performance characteristics: Single-pass, streaming, O(n) in file size. The
  seen map for dedup is bounded by the number of unique message IDs in the
  session (typically hundreds, not thousands). No full parse into turn pairs ‚ÄĒ
  just line-by-line extraction.

‚óŹ Section 5: Hook Integration

  Capy's hook system currently handles PreToolUse events. Vault adds two new
  event types: PreCompact and Stop (session end).

  PreCompact is the critical path ‚ÄĒ it fires before /compact rewrites the
  session file. The hook:
  1. Receives session file path from the event payload (stdin JSON)
  2. Reads the raw JSONL from disk (before compaction destroys it)
  3. Writes to vault_snapshots (cold storage, append-only ‚ÄĒ preserves
  pre-compaction content)
  4. Writes/replaces vault_sessions + FTS rows (active state ‚ÄĒ subject to
  size-based merge logic)
  5. Discovers and archives subagent files if present

  Stop (session end) fires when a session completes. Lower priority than
  PreCompact but keeps the vault fresh without requiring explicit import runs.
  Same archival logic as import: idempotent, size-based merge, transactional.

  Hook wiring: Both events route through internal/hook/ with direct calls into
  the vault package ‚ÄĒ no dynamic registry. The hook router switches on event
  type:
  - PreCompact, Stop ‚Üí vault.ArchiveSession(payload)
  - PreToolUse ‚Üí existing logic

  Environment: Hooks run as short-lived capy hook <event> subprocesses.
  CAPY_VAULT_KEY must be available in the shell environment (exported in
  .zshrc/.bashrc). The hook process opens vault.db, does its work, and exits.
  Each invocation is independent ‚ÄĒ no long-lived connection pool.

  Metadata from execution context: Because hooks execute inside the project
  directory, they can capture accurate metadata that bulk import cannot:
  - os.Getwd() ‚Üí accurate project path (no unmangling needed)
  - git rev-parse --abbrev-ref HEAD ‚Üí current branch

  Failure mode: If the hook fails (vault DB locked, missing key, disk full), it
  logs a warning and exits non-zero. It must not block Claude Code's operation ‚ÄĒ
   the session continues regardless. The session can be recovered later via capy
   vault import.

‚óŹ Section 8: Cross-Machine Merge

  The cross-machine workflow: copy vault.db from machine A to machine B, run
  capy vault import on B, and get sessions from both machines without duplicates
   or data loss.

  How it works in practice:

  1. User copies vault.db from machine A ‚Üí machine B (replaces B's existing
  vault.db, or B has no vault yet)
  2. On machine B, run capy vault import ‚ÄĒ this scans B's local
  ~/.claude/projects/*/
  3. B's local sessions get inserted into the vault that already contains A's
  sessions
  4. UUID-based dedup prevents duplicates. Size-based merge keeps the fuller
  version.

  Merge behavior per table:

  - vault_sessions: If UUID doesn't exist ‚Üí insert. If UUID exists with same
  content_hash ‚Üí skip. If UUID exists with different hash ‚Üí keep larger
  size_bytes version (replace entire row within a transaction: delete old FTS
  rows, insert new session + FTS rows).
  - vault_subagents: Cascades with parent session via ON DELETE CASCADE. When a
  session is replaced, its subagents are replaced too.
  - vault_snapshots: INSERT OR IGNORE on (uuid, content_hash). Snapshots from
  both machines accumulate ‚ÄĒ append-only, never overwritten.
  - vault_fts: Rebuilt as part of session insert/replace transactions. No
  independent merge needed.

  Machine provenance: Each session carries a machine_id (UUIDv4 stored in
  vault_meta). After copying vault.db from A to B, B's vault_meta.machine_id
  gets overwritten with A's value. On first import, B regenerates its own
  machine_id (since the meta table already has one from A).

  Actually ‚ÄĒ that's a problem. The vault_meta machine_id would be A's after the
  copy. We should handle this: on import, if the current OS machine identity
  differs from what's in vault_meta, generate a new machine_id and store it
  alongside (or just let new sessions get tagged with the new ID). The simplest
  approach: vault_meta stores the local machine_id, and it's regenerated if the
  DB was clearly copied from elsewhere (e.g., a capy vault init flag, or just
  always generate on first local import if no local sessions exist yet).

  Simpler: don't store machine_id in vault_meta at all. Instead, compute it at
  runtime from a local file ~/.config/capy/machine-id (generated once, persists
  across DB copies). This way, copying vault.db doesn't carry the machine
  identity ‚ÄĒ each machine has its own stable ID independent of the database.

  Does this revised approach to machine_id look right?
