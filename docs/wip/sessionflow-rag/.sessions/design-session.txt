  ---
  Section 1: Overview and Problem Statement

  Claude Code stores conversation sessions as JSONL files in
  ~/.claude/projects/<project-dir-name>/. These sessions contain valuable
  decisions, rationale, and context that is currently lost after conversations
  end. capy already solves context flooding and provides persistent queryable
  memory, but only for content explicitly indexed during a session ‚ÄĒ the
  conversations themselves are invisible to future searches.

  Sessionflow RAG makes past sessions automatically searchable by indexing their
   human/assistant text content into capy's FTS5 knowledge base. A background
  sweep runs at MCP server startup, parsing session JSONL files, filtering out
  noise (tool results, system messages, trivial sessions), chunking the
  conversational text into BM25-friendly windows, and indexing them under a new
  KindSession source kind. Sessions are included in default search results and
  have a configurable TTL (default 60 days) to preserve "progress over time" for
   the project.

  The feature is entirely automatic ‚ÄĒ no MCP tool is exposed, the assistant
  never needs to think about it, and the indexed content stays portable across
  machines by using machine-agnostic labels (session UUID + date, not file
  paths).

  ---
  Section 2: Session File Parsing and Filtering

  Session JSONL files contain one JSON object per line. Each line has a type
  field: user, assistant, attachment, file-history-snapshot, permission-mode,
  last-prompt. Only user and assistant types carry conversation content.

  User messages have message.content as either a plain string or an array of
  content blocks. We extract only string content and {"type": "text"} blocks.
  tool_result blocks are dropped entirely ‚ÄĒ they contain raw tool output (file
  contents, command output) that would pollute BM25 term frequencies. Human
  messages starting with / (slash commands like /config, /compact, /help) are
  skipped.

  Assistant messages have message.content as an array of blocks. We extract
  {"type": "text"} blocks only. tool_use blocks are not included in the
  transcript text, but we extract the name field from each for metadata (tool
  names appear in chunk titles for searchability). thinking blocks are skipped ‚ÄĒ
   they're internal reasoning and not part of the visible conversation.

  Compact summaries ‚ÄĒ when Claude Code compacts a conversation, the resulting
  summary message is dense, high-signal assistant text. These are kept and
  treated as regular assistant text. They're excellent for BM25.

  Session-level gate: after filtering, a session must have at least 2 meaningful
   turn pairs (human + assistant text response) AND at least 200 characters of
  total assistant text. Sessions that don't pass are skipped entirely ‚ÄĒ this
  catches config-only sessions, quick one-off commands, and aborted
  conversations.

  Section 3: Chunking Strategy

  Session transcripts are converted to a plaintext format before chunking. Each
  turn becomes:

  Human: <text>
  Assistant: <text>
  [Tools: Read, Edit, Bash]

  Tool-result-only user messages are omitted. system/away_summary entries are
  inserted inline as [Session summary: <text>]. Sub-agent conversations are
  delimited with --- Subagent: <type> ‚ÄĒ "<description>" --- and --- End subagent
   --- markers.

  Chunking uses a sliding window of ~4 turn pairs with 1-pair overlap, adapted
  from chunkPlainText. The split happens on turn boundaries (the Human: prefix),
   not arbitrary line counts. This preserves conversational coherence ‚ÄĒ a
  question and its answer always stay together.

  Each chunk gets a title: Session 2026-04-05T12:06:26Z | Turns 3-6 | Tools: 
  Read, Edit. When the window contains sub-agent turns, the title appends | 
  Subagent: Explore. MCP tool names are normalized from mcp__capy__capy_search
  to capy:search for readability.

  Chunks exceeding maxChunkBytes (4096) are split at paragraph boundaries using
  the existing splitOversized logic. The overlap ensures continuity: if a
  decision spans two chunks, the overlapping turn appears in both, so BM25 can
  surface either.

  The content type is "session" ‚ÄĒ a new value added to the content type taxonomy
   alongside "code", "prose", "plaintext", "markdown", "json".

‚óŹ ---
  Section 5: Session Discovery and Sweep Mechanism

  Session indexing is triggered at MCP server startup as a background goroutine.
   The sweep runs once per server boot ‚ÄĒ not continuously.

  Discovery: Claude Code stores sessions in ~/.claude/projects/<dir-name>/ where
   <dir-name> is the absolute project path with / replaced by - and leading -
  (e.g. -home-sergio-Projects-personal-capy). The server already knows
  projectDir ‚ÄĒ it derives the session directory by mangling that path the same
  way Claude Code does, then verifying the directory exists.

  Sweep algorithm:
  1. List all .jsonl files in the session directory
  2. For each file, stat() to get mtime
  3. Query the store for a source with label session:<datetime>:<uuid> ‚ÄĒ but
  since we don't know the datetime before parsing, we need a different mtime
  gate. Instead: query all sources with session: prefix, build a map of uuid ‚Üí 
  indexed_at. Compare file mtime against indexed_at. If mtime <= indexed_at,
  skip.
  4. For files that pass the mtime gate: parse JSONL, apply filtering, build
  transcript, compute content_hash
  5. Call Index() with the session label, content type "session", and
  KindSession. Existing dedup handles the rest.
  6. For sessions with sub-agent directories (<uuid>/subagents/*.jsonl): parse
  sub-agent files and append to the parent transcript before chunking.

  Current session exclusion: The session that just started will have a
  near-empty JSONL file. It passes through the sweep but gets filtered out by
  the session-level gate (< 2 turn pairs).

  Error handling: Parse failures on individual sessions are logged and skipped ‚ÄĒ
   one corrupt file doesn't block the sweep. The goroutine has a timeout (30s
  default) to avoid blocking startup indefinitely.
